---
title: Row Diff Engine
description: Internal notes on how Graft produces SQLite row diffs and merge diagnostics.
---

The row diff engine compares SQLite snapshots and attempts to describe changes
in logical table terms. It exists so repository diffs and merge UI can show more
than "the database file changed."

## Public Entry Points

Use repository-level row diff for product integrations:

```bash
graft diff --rows HEAD~1 HEAD data.sqlite
graft diff --json --rows HEAD~1 HEAD data.sqlite
```

SQLite:

```sql
PRAGMA graft_diff = '--rows HEAD~1 HEAD -- data.sqlite';
PRAGMA graft_json_diff = '--rows HEAD~1 HEAD -- data.sqlite';
```

## Debug Entry Points

Low-level storage diagnostics can compare raw storage LSNs:

```sql
PRAGMA graft_debug_log_lsn;
PRAGMA graft_debug_volume_diff = '2,4,rows';
PRAGMA graft_debug_volume_json_diff = '2,4,rows';
```

These are for implementation work and storage debugging. They should not be the
normal application API.

## Output Shape

JSON row diff output can include:

```json
{
  "path": "data.sqlite",
  "change": "modified",
  "row_diff_available": true,
  "logical_status": "logical_changes",
  "capabilities": ["rowid_table_rows", "schema_entries"],
  "limitations": [],
  "tables": []
}
```

`logical_status` lets applications distinguish supported logical row changes
from conservative or unavailable cases.

## Merge Diagnostics

During merge, the same analysis can produce:

- row conflicts
- schema conflicts
- opaque changes
- resolved internal changes
- limitations
- blocked reasons
- apply policy and validation details

This output feeds `graft conflicts --json` and `PRAGMA graft_json_conflicts`.

## Resolution Layer

The row diff engine works below repository revspecs. Repository commands resolve
`HEAD`, branches, tags, and commit ids into the snapshot objects the row engine
can compare. Applications should keep using repository-level commands so the UI
stays aligned with branches, refs, and path state.
