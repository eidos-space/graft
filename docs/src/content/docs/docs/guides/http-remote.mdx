---
title: Connect An HTTP Remote
description: Connect Graft v0.6.0 to a server that implements the Graft HTTP remote protocol.
---

Graft v0.6.0 includes an HTTP remote client. It can fetch and push repository
objects through a service that implements the Graft HTTP remote protocol.

Graft does **not** bundle or deploy that server. You need an existing compatible
service or your own implementation of the protocol described in
[HTTP Remote Protocol](/docs/internals/http-remote-protocol/).

## Configure A Remote

Set the bearer token in the process environment, then add the remote:

```bash
export GRAFT_REMOTE_TOKEN='grt_...'
graft remote add origin \
  'graft+https://example.com/api/graft/v1/repos/acme/archive'
```

The `graft+https://` prefix selects the Graft HTTP transport. The underlying
request URL uses normal HTTPS.

Use `token_env` when the service token lives in a different variable:

```bash
export GRAFT_ARCHIVE_TOKEN='grt_...'
graft remote add origin \
  'graft+https://example.com/api/graft/v1/repos/acme/archive?token_env=GRAFT_ARCHIVE_TOKEN'
```

Do not put the token itself in the URI or `.graft/config.toml`.

## Verify And Sync

Inspect the stored URL and advertised refs before pushing:

```bash
graft remote get-url origin
graft ls-remote origin
graft push origin main
```

On another worktree:

```bash
graft clone \
  'graft+https://example.com/api/graft/v1/repos/acme/archive' \
  main
```

The clone process needs the same token environment variable.

## Local Protocol Development

Use `graft+http://` only for a local server you control:

```bash
export GRAFT_REMOTE_TOKEN='dev-token'
graft remote add origin \
  'graft+http://127.0.0.1:8787/api/graft/v1/repos/acme/archive'
```

For production, use HTTPS and let the service enforce repository authorization,
object integrity, request limits, and durable storage.

See [Remote URIs](/docs/reference/remote-uris/) for URI syntax and
[Sync With Remotes](/docs/guides/sync-remotes/) for branch workflows.
